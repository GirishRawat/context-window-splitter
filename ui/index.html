The six-phase compilation pipeline

LLM-optimized LLVM IR, made valid by construction: every function in the output is either a formally-proven refinement of the original, or the untouched -O0 original itself. All probabilistic risk lives in Phase 3 and is fully contained by the deterministic verification gate around it.

Deterministic Probabilistic (the only one) Verification gate
  1. 1

    Parsing & Extraction

    Deterministic

    Ingests -O0 IR, verifies it, and extracts every function definition into a standalone, independently-assemblable unit (shared preamble + sibling declarations + body).

    Implemented → FunctionRecord.original_ir Live view: Phase 1 tab
  2. 2

    Triage & Profiling

    Deterministic

    Scores cyclomatic complexity and token count per function. Below-threshold (too simple) functions are triaged out — they skip optimization and pass straight through unchanged.

    Implemented → complexity, token_count, triaged_out Live view: Phase 2 tab
  3. 3

    LLM Execution & Routing

    Probabilistic

    Routes each non-triaged function to a local Ollama model by complexity/token tier and runs it as a stateless, single-turn text-in / text-out transformation. No memory, no feedback loops, no tool use. Zero API cost — all inference runs on-device.

    Ollama Integration → llm_output (a define block) Model: qwen2.5-coder (3b / 7b)
  4. 4

    Candidate Reconstruction

    Deterministic

    Substitutes the candidate body back into the function's own standalone context — keeping the same preamble and sibling declarations — so the gate compares like against like.

    Implemented → candidate_ir
  5. 5

    Verification Gate

    Deterministic · Gate

    llvm-as syntax check, then an alive-tv (Alive2/Z3) refinement proof: source = original, target = candidate. PASSED only if proven; a timeout is never a pass. This is the single point that guarantees correctness.

    Implemented Needs M0 toolchain to run → verdict, counterexample Live view: Phase 5 tab
  6. 6

    Fallback Assembly

    Deterministic

    PASSED → the proven optimization is locked in. Anything else (rejected, unsupported, syntax-fail, triaged out) → the untouched original is reinserted from memory. Then the final module is assembled.

    Implemented → final_ir, final_module_ir Compile-to-executable: deferred (needs clang)

Where the build is now

Milestone 1 (walking skeleton) is complete — the full 1→6 spine runs deterministically end-to-end with Phase 3 as an identity transform, proving all plumbing with zero API calls. Because Phase 5 needs the external llvm-as + alive-tv toolchain (Milestone 0), a real run currently fails closed: every function falls back to its original, so the output is behaviourally identical to an unoptimised build — the correct, safe outcome. Next up: M0, building the Z3 / LLVM / Alive2 toolchain so the gate can prove refinements for real.
